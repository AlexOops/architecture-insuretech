# Task1 — Проектирование технологической архитектуры

## Цель

Спроектировать технологическую архитектуру InsureTech так, чтобы она соответствовала бизнес-требованиям:

- 24/7 доступность;
- SLA / доступность 99.9%;
- RTO = 45 минут;
- RPO = 15 минут;
- одинаковое время загрузки страниц для пользователей из разных регионов РФ;
- готовность к росту нагрузки.

## Ключевые архитектурные решения

### 1. Масштабирование
- Для приложений выбрано **горизонтальное масштабирование**:
  - stateless-сервисы масштабируются через HPA;
  - node groups масштабируются через Cluster Autoscaler.
- Для PostgreSQL выбрано **вертикальное масштабирование + репликация**, без шардирования.

### 2. Kubernetes
- Выбран **один региональный Kubernetes cluster**, растянутый на **3 зоны доступности**.
- Причина выбора:
  - проще сопровождение;
  - единая control plane;
  - встроенная отказоустойчивость между AZ;
  - меньше сложность, чем у независимых кластеров по зонам.

### 3. Балансировка нагрузки
- Для частных пользователей:
  - DNS → CDN → WAF/DDoS → Application Load Balancer → Ingress → сервисы.
- Для B2B и партнёров:
  - DNS → API Gateway → WAF/DDoS → Application Load Balancer → Ingress → сервисы.
- API Gateway используется для:
  - rate limiting;
  - quota;
  - auth;
  - защиты SLA от шумных партнёров.

### 4. Равномерное время загрузки страниц
- Статический frontend и ассеты размещаются в Object Storage и раздаются через CDN.
- Это снижает зависимость времени загрузки страниц от географии пользователя.

### 5. База данных
- PostgreSQL развёрнут в отказоустойчивой конфигурации:
  - Primary — AZ-A
  - Synchronous Standby — AZ-B
  - Asynchronous Replica — AZ-C
- Для автоматического failover используется Patroni (или managed failover).
- Настроены:
  - регулярные бэкапы;
  - WAL archiving;
  - PITR.

### 6. Шардирование
- **Не используется**, потому что:
  - объём данных всего 50 GB;
  - нагрузка не требует шардирования;
  - сложность шардирования на данном этапе неоправданна.

## Соответствие требованиям

- **99.9% availability** — достигается за счёт multi-AZ deployment, LB, HA PostgreSQL.
- **RTO 45 min** — достигается за счёт автоматического failover и подготовленной standby-реплики.
- **RPO 15 min** — достигается за счёт synchronous replication внутри региона и WAL archiving / backup strategy.
- **Одинаковое время загрузки страниц** — достигается за счёт CDN.
