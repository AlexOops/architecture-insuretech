# Task4 — Проектирование продажи ОСАГО

## Цель

Спроектировать архитектуру продажи ОСАГО онлайн с учётом требований:

- пользователь создаёт заявку на ОСАГО;
- сервис отправляет заявку во все доступные страховые компании;
- предложения отображаются пользователю сразу по мере получения;
- максимальное время ожидания ответа от страховой компании — 60 секунд;
- пиковая нагрузка — до 2500 одновременных пользователей;
- сервисы развёрнуты в нескольких экземплярах.

## Ключевые решения

### 1. Новый сервис osago-aggregator

Добавляется отдельный сервис `osago-aggregator`.

Его ответственность:

- принимать команды на расчёт ОСАГО;
- отправлять заявки в страховые компании;
- хранить состояние запросов к каждой страховой компании;
- опрашивать страховые компании по external application id;
- публиковать предложения по мере получения;
- завершать расчёт через 60 секунд или раньше, если все страховые компании ответили.

### 2. Хранилище osago-aggregator

`osago-aggregator` требуется собственное хранилище данных.

Причины:

- процесс расчёта длится до 60 секунд;
- сервис работает в нескольких экземплярах;
- нельзя хранить состояние расчёта в памяти pod'а;
- pod может быть перезапущен во время расчёта;
- нужно хранить external application id для каждой страховой компании;
- нужны retry_count, status, deadline_at и результат по каждой страховой компании.

Пример данных:

- quote_request_id;
- insurance_company_id;
- external_application_id;
- status;
- retry_count;
- next_poll_at;
- deadline_at;
- offer payload;
- error reason.

### 3. Интеграция core-app и osago-aggregator

Для связи `core-app` и `osago-aggregator` используется Event Streaming.

Топики:

- `osago-quote-requests.v1` — команда на расчёт ОСАГО;
- `osago-offers.v1` — предложения и статусы расчёта;
- `osago-offers.dlq.v1` — ошибки обработки.

`core-app` публикует событие `OsagoQuoteRequested`.

`osago-aggregator` публикует события:

- `OsagoOfferReceived`;
- `OsagoCompanyTimedOut`;
- `OsagoQuoteCompleted`;
- `OsagoQuoteFailed`.

### 4. API core-app для веб-приложения

Для веб-приложения используется комбинированный подход:

1. REST — для создания заявки:
   - `POST /api/osago/quotes`

2. SSE — для получения предложений в реальном времени:
   - `GET /api/osago/quotes/{quoteRequestId}/events`

Почему SSE:

- предложения идут только от сервера к клиенту;
- нужно показывать результат сразу по мере получения;
- SSE проще WebSocket для однонаправленного потока событий;
- поддерживает reconnect;
- можно использовать `Last-Event-ID`.

### 5. Работа при нескольких репликах

Решение не зависит от конкретного pod'а.

- состояние заявки хранится в БД;
- события передаются через Kafka;
- `osago-aggregator` хранит состояние polling в своей БД;
- `core-app` хранит read model предложений;
- SSE может восстановить состояние из БД после reconnect.

Для предотвращения двойной обработки используются:

- idempotency key;
- event_id;
- inbox table у потребителей;
- transactional outbox у издателей событий.

## Паттерны отказоустойчивости

### Web → core-app

- Rate Limiter — защита от всплесков трафика;
- Timeout — ограничение времени обработки REST-запроса;
- SSE reconnect — восстановление соединения.

### core-app → Kafka

- Transactional Outbox;
- Retry публикации;
- DLQ для невалидных сообщений.

### osago-aggregator → страховые компании

Для каждой страховой компании отдельно применяются:

- Rate Limiter;
- Circuit Breaker;
- Retry with backoff;
- Timeout;
- общий deadline 60 секунд на получение предложения.

### osago-aggregator → Kafka

- Transactional Outbox;
- Retry;
- DLQ.

## Почему не REST между core-app и osago-aggregator

REST не подходит как основной способ интеграции, потому что расчёт ОСАГО длится до 60 секунд и включает fan-out во все страховые компании.

Если держать синхронный HTTP-запрос открытым:

- увеличивается риск timeout;
- растёт нагрузка на core-app;
- сложнее масштабировать сервисы;
- сложнее восстанавливаться после перезапуска pod'а.

Поэтому core-app только инициирует расчёт, а результаты получает асинхронно через события.
