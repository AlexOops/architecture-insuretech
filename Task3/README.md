# Task3 — Переход на Event-Driven архитектуру

## Цель

Проанализировать текущую архитектуру взаимодействия сервисов `core-app`, `ins-product-aggregator` и `ins-comp-settlement`, выявить проблемы при росте нагрузки и предложить переход на Event-Driven архитектуру без изменения функциональной декомпозиции сервисов.

## Текущая схема взаимодействий

Сейчас используются синхронные REST-вызовы и периодический polling:

1. `core-app` раз в 15 минут запрашивает продукты и тарифы у `ins-product-aggregator`.
2. `ins-comp-settlement` раз в сутки ночью запрашивает продукты и тарифы у `ins-product-aggregator`.
3. `ins-product-aggregator` в момент REST-вызова обращается к API всех страховых компаний, агрегирует результат и возвращает его в рамках того же синхронного запроса.
4. `ins-comp-settlement` раз в сутки запрашивает у `core-app` список всех оформленных за день страховок.

## Проблемы и риски текущей архитектуры

### 1. Синхронный fan-out к страховым компаниям

`ins-product-aggregator` при одном входящем запросе обращается сразу ко всем страховым компаниям.

Сейчас страховых компаний 5, но планируется добавить ещё 5. Количество внешних вызовов вырастет в 2 раза.

Риски:

- рост latency;
- рост количества timeout-ошибок;
- зависимость ответа от самой медленной страховой компании;
- каскадные отказы;
- увеличение нагрузки на `ins-product-aggregator`.

### 2. Периодический polling

`core-app` и `ins-comp-settlement` регулярно опрашивают `ins-product-aggregator`.

Риски:

- лишние запросы, даже если данные не изменились;
- неравномерная нагрузка;
- возможная устарелость локальных копий данных;
- задержка доставки изменений до 15 минут или до суток;
- сложность контроля актуальности локальных реплик.

### 3. Ночная batch-нагрузка на settlement

`ins-comp-settlement` раз в сутки получает все оформленные страховки из `core-app`.

Риски:

- большой batch-запрос;
- высокая нагрузка на `core-app` и БД;
- если ночной процесс упал, данные за день могут быть обработаны с задержкой;
- нет near real-time обновления данных для взаиморасчётов.

### 4. Жёсткая связанность сервисов

`ins-comp-settlement` зависит от доступности `core-app`, а `core-app` и `ins-comp-settlement` зависят от доступности `ins-product-aggregator`.

Риски:

- сбой одного сервиса влияет на другие;
- сложнее выполнять независимые релизы;
- сложнее масштабировать сервисы независимо.

### 5. Нет встроенной стратегии retry / DLQ

При ошибках внешних API или внутренних REST-вызовов нет гарантированного механизма повторной обработки.

Риски:

- потеря данных;
- ручные перезапуски задач;
- сложная диагностика инцидентов.

### 6. Нет надёжной доставки изменений

Если данные о продуктах, тарифах или оформленных страховках изменились, потребители получают их только при следующем polling-запросе.

Риски:

- локальные реплики расходятся;
- сложно гарантировать консистентность;
- возможны ошибки при расчёте агентских премий.

## Целевая архитектура

Предлагается добавить Event Streaming Platform, например Kafka, и перевести часть взаимодействий на событийную модель.

Функциональная декомпозиция сервисов не меняется:

- `core-app` по-прежнему отвечает за оформление страховок и клиентский пользовательский сценарий;
- `client-info` по-прежнему отвечает за клиентские данные;
- `ins-product-aggregator` по-прежнему отвечает за интеграцию со страховыми компаниями и нормализацию продуктов;
- `ins-comp-settlement` по-прежнему отвечает за взаиморасчёты со страховыми компаниями.

Меняется только способ обмена данными между сервисами.

## Какие взаимодействия переводятся на Event Streaming

### 1. Данные о продуктах и тарифах

Вместо polling-запросов от `core-app` и `ins-comp-settlement` сервис `ins-product-aggregator` публикует события:

- `ProductCatalogUpdated`
- `ProductTariffUpdated`
- `ProductDisabled`
- `InsuranceCompanySyncFailed`

Топик:

```text
insurance-products.v1
```

Потребители:

- `core-app`
- `ins-comp-settlement`

Каждый потребитель обновляет свою локальную read model продуктов и тарифов.

### 2. Данные об оформленных страховках

Вместо ночного REST-запроса от `ins-comp-settlement` к `core-app`, сервис `core-app` публикует события:

- `PolicyCreated`
- `PolicyUpdated`
- `PolicyCancelled`
- `PolicyIssued`

Топик:

```text
policies.v1
```

Потребитель:

- `ins-comp-settlement`

`ins-comp-settlement` постепенно накапливает данные для реестра взаиморасчётов и больше не зависит от ночного batch-запроса в `core-app`.

## Что остаётся через REST

REST остаётся там, где нужен синхронный пользовательский сценарий:

- `InsureTech Web` → `core-app`;
- `core-app` → `client-info`;
- внешние API страховых компаний для `ins-product-aggregator`.

Вызовы к страховым компаниям остаются внешней интеграцией, но их результат больше не должен запрашиваться синхронно в момент пользовательского запроса. `ins-product-aggregator` может обновлять каталог по расписанию, webhook-уведомлениям или внутренним integration jobs.

## Transactional Outbox

Для надёжной публикации событий используется паттерн Transactional Outbox.

### Где используется Outbox

1. `core-app`
   - записывает оформленную страховку в свою БД;
   - в той же транзакции пишет событие в таблицу `outbox`;
   - outbox relay публикует событие в Kafka.

2. `ins-product-aggregator`
   - сохраняет обновлённый каталог продуктов и тарифов в свою локальную БД;
   - в той же транзакции пишет событие в таблицу `outbox`;
   - outbox relay публикует событие в Kafka.

### Почему нужен Outbox

Без Outbox возможна ситуация:

- данные в БД записались, но событие не отправилось;
- событие отправилось, но транзакция в БД откатилась.

Transactional Outbox устраняет этот разрыв и гарантирует, что событие соответствует состоянию данных в БД.

## Inbox / Idempotent Consumer

Так как Kafka и outbox relay могут доставить событие повторно, потребители должны быть идемпотентными.

Для этого в `core-app` и `ins-comp-settlement` используется Inbox-подход:

- у каждого события есть `event_id`;
- потребитель хранит обработанные `event_id`;
- повторное событие не изменяет состояние второй раз.

## Event keys и порядок событий

Для корректного порядка обработки событий используются ключи:

- для продуктов: `product_id` или `insurance_company_id + product_id`;
- для тарифов: `tariff_id`;
- для страховок: `policy_id`.

Это позволяет Kafka сохранять порядок событий внутри одной сущности.

## Retry и DLQ

Для повышения надёжности используются:

- retry при временных ошибках;
- dead-letter topic для событий, которые не удалось обработать;
- алерты на рост DLQ;
- ручной reprocess после исправления причины ошибки.

Примеры DLQ-топиков:

```text
insurance-products.dlq.v1
policies.dlq.v1
```

## Schema Registry

Для контроля совместимости событий рекомендуется использовать Schema Registry.

Это снижает риск поломки потребителей при изменении структуры события.

## Ожидаемый эффект

### Для core-app

- больше не нужно раз в 15 минут опрашивать `ins-product-aggregator`;
- локальная копия продуктов обновляется через события;
- меньше зависимость от задержек API страховых компаний.

### Для ins-comp-settlement

- больше не нужно ночью запрашивать все страховки из `core-app`;
- данные для взаиморасчётов поступают постепенно;
- снижается нагрузка на `core-app`;
- уменьшается риск падения ночного batch-процесса.

### Для ins-product-aggregator

- сервис перестаёт быть синхронным узким местом для внутренних потребителей;
- интеграция со страховыми компаниями становится изолированной;
- ошибки отдельных страховых компаний не ломают пользовательский сценарий напрямую.

## Итоговое решение

В целевой архитектуре используются:

- Kafka / Event Streaming Platform;
- топик `insurance-products.v1`;
- топик `policies.v1`;
- DLQ-топики;
- Transactional Outbox для `core-app` и `ins-product-aggregator`;
- Inbox / Idempotent Consumer для потребителей;
- Schema Registry для контроля схем событий.

Такое решение снижает связанность сервисов, убирает polling, уменьшает влияние задержек внешних API и позволяет системе устойчиво масштабироваться при подключении новых страховых компаний.
